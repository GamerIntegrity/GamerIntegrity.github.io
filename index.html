What it checks

GamerIntegrity looks for local traces that can help staff understand what happened on a PC.

Launch
Was something run?

Looks for traces of loaders, injectors, mappers, spoofers, and other suspicious tools being opened.

Downloads
What was searched or downloaded?

Checks browser visit and download records tied to cheat sites, cheat terms, local paths, source pages, and tools.

Projects
Source and builds

Looks for cheat projects, pasted sources, offsets, SDKs, and build folders.

Startup
What starts with Windows?

Checks running apps, services, drivers, scheduled tasks, and startup entries.

Files
Names and folders

Looks for local file and folder names tied to direct cheat terms, cheat tools, or cheat projects.

Context
USB, limitations, browser, hardware

Shows extra context that can help explain movement, timing, failed reads, or the PC being checked.

Known cheat types we scan for

GamerIntegrity looks for traces tied to common cheat types and cheat-tool ecosystems. One weak word is not the whole story. Stronger results usually come from several areas pointing the same way.

Areas checked for these cheat types

GamerIntegrity checks local Windows and browser traces that can help show whether cheat tools were searched, downloaded, stored, built, launched, opened recently, or left behind.

  • Direct launch traces — Prefetch and app-history style evidence for tools that were run.
  • Recent app activity — UserAssist and BAM/DAM history that can show tools opened through Explorer or retained by Windows.
  • Retained history — Jump Lists, Recent Items, ShellBags, Run box history, RecentDocs, and Open/Save dialog history.
  • Downloads — browser visit/download records tied to cheat sites, providers, tools, local paths, source links, or cheat wording.
  • Startup items — running processes, services, drivers, tasks, and startup entries.
  • Projects — source folders, pasted bases, offsets, SDK dumps, build outputs, and cheat-tool projects.
  • Files / folders — names and paths that match known cheat terms, tools, providers, or suspicious project layouts.
  • Browser history — Chrome, Edge, Brave, Firefox, cheat sites/domains, provider domains, direct cheat terms, game-specific cheat terms, and download pages.
  • Security history — Windows Defender history, selected event logs, and cleanup/missing-trace hints when available.
  • SRUM activity — extra app-activity context when Windows exposes useful app/path data.
  • External devices — USB and external device history that may help explain timing or device swapping.
  • Hardware — PC identity context. One scan alone does not prove spoofing, but it helps staff understand the setup.
Fair check note: GamerIntegrity looks for traces. Staff should still review timing, context, and stronger evidence before making a decision.

JSON

GamerIntegrity_Report_Integrity.json

Contains the report hashes so reviewers can check whether the HTML or JSON changed after export.

Best practice: share the redacted HTML for normal review, but preserve the HTML, JSON, and integrity manifest together.

Fair play

GamerIntegrity is not an instant ban button. It helps show what was found on a PC so staff can make a better call. Strong decisions should come from chains of evidence, not one weak row.
  • Old downloads, dev tools, browser history, report pages, or test folders can show up in a scan.
  • Some tools are dual-use and need context.
  • Strong results usually have multiple traces pointing the same way.
  • Each community should follow its own rules and appeal process.

FAQ

Does GamerIntegrity upload my files?

No. Scans are local. Sharing only happens if someone exports and sends a report.

Does it need an account?

No. GamerIntegrity is non-gate kept and does not require a login to run a check.

Why does it ask for admin?

Some Windows traces need admin access. The app shows Restart as Admin when needed.

Does it open the browser after every scan?

No. Results load inside GamerIntegrity first. Files are only created when exported.

Which report should I share?

Use Redacted for sharing. Use Non-Redacted only for a trusted local check. Keep the HTML, JSON, and integrity file together when a report needs to be reviewed or preserved.

Does a Critical result automatically prove cheating?

No. It means strong traces were found. Staff should check the categories, timeline, direct launches, downloads, local files, and context before making a decision.

Can normal tools be flagged?

Yes. Some developer, debugging, or reverse-engineering tools can be legitimate. Treat those as context unless stronger cheat-related traces are also present.

Why do I see a Windows SmartScreen warning?

New or unsigned community apps can trigger SmartScreen. Download only from the official GitHub release page.

Can someone delete traces before a scan?

Some traces can be missing or cleaned. That is why staff should review the full picture instead of one field.

Where do I report bugs?

Use the official GitHub repo so issues and fixes can be tracked publicly.
